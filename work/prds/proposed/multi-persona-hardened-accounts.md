## Confirmed decisions

All decisions are settled; `needsAnswers` is cleared and the PRD is ready to task. Durable rationale to move into an ADR at tasking (this PRD SUPERSEDES the single-account framing of `docs/adr/0006` — a new ADR extends/replaces it).

0. **Two formerly-open items, now DECIDED:**
   - **subuid/subgid allocation = let `useradd` auto-allocate.** anon-pi emits NO explicit `/etc/subuid`+`/etc/subgid` range line in the generated Tier-2 script; it trusts `useradd -m` to find a free non-overlapping block (modern shadow-utils via `SUB_UID_COUNT`/`SUB_GID_COUNT` in `/etc/login.defs`). This removes the range-collision math from anon-pi entirely. v1's single-account generator (`hardened-tier2-script-generator`, `SUBID_RANGE_START`/`SUBID_RANGE_COUNT`) is UPDATED to match (drop the hard-coded range), so the default `anon` and every additional persona use one uniform auto-allocation path. The preflight's "subuid/subgid ranges present" check still verifies a range EXISTS for the account (it just no longer asserts a specific start).
   - **`anon` is uniform "persona #0", not special-cased.** `anon` is simply the DEFAULT-named persona, created by the same `persona add` path as any other. An existing v1 hardened install is a multi-persona install with exactly one persona named `anon`. No distinguished always-present account; the code stays one shape (a list of personas, one of which is the default name `anon`).

1. **Multiple dedicated persona accounts, `anon` is the default.** Instead of one shared `anon` account (v1), support N dedicated Unix accounts, each its own unlinkable persona with its own mode-700 home, session history, netcage store, AND egress. `anon` remains the default persona name so existing v1 hardened installs keep working unchanged. This SUPERSEDES ADR-0006's "single dedicated account named `anon`" decision.

2. **Selection = interactive TYPED ask, defaulting to `anon`; `--as <name>` is the opt-in convenience.** On a multi-persona install, when a hardened invocation is made, anon-pi ASKS which persona and the user TYPES the name (or accepts the `anon` default). RATIONALE (load-bearing): the persona name is a sensitive token — typing it as a positional/flag would write it into shell history, `ps`, and audit logs, re-associating the persona with the operator. An interactive prompt keeps the name OUT of argv/history. A `--as <name>` flag exists for scripts/non-TTY convenience and is the explicit "I accept the name may be logged" escape hatch. A typed name that matches no known persona RE-ASKS (typo), never silently creates.

3. **Per-persona FAIL-CLOSED egress = one socks5h endpoint per persona.** Each persona has its OWN egress; no persona shares another's exit. Fail-closed is per persona: a persona with no resolvable proxy REFUSES to launch, byte-identical to today's `PROXY_REQUIRED_MESSAGE`. Never falls back to another persona's proxy or to no proxy. Netcage's forced-egress invariant is UNCHANGED: still one socks5h endpoint forced per launch, still never guessed. This feature only changes WHICH endpoint (per persona) and WHICH user anon-pi runs as; it never weakens the jail. NO rotation knob (a persona = one endpoint; Tor's natural circuit churn provides coarse drift for free, which is enough).

4. **Two ways to get a per-persona endpoint: Tor multi-persona (preferred) or bring-your-own SOCKS.**
   - **Tor multi-persona:** if `persona add` DETECTS a running Tor (probe the SOCKS port, reusing init's existing proxy probe), offer it. anon-pi uses the persona's ACCOUNT USERNAME as the Tor SOCKS-isolation username → `socks5h://<account>:x@127.0.0.1:9050`. Tor's `IsolateSOCKSAuth` (on by default) then builds a SEPARATE circuit + exit per distinct SOCKS username, so each persona gets an independent, auto-managed, auto-expiring circuit (idle circuits tear down at `MaxCircuitDirtiness` ~10 min and rebuild on next use) at near-zero marginal cost (a circuit is state in the one daemon, not a process). `persona add` composes this literal URL once (account name as the isolation username) and stores it in the persona's own `config.json` `proxy` field (see decision 5) — no launch-time derivation, no schema marker.
   - **Bring-your-own SOCKS:** otherwise the user picks a socks5h endpoint they run (a distinct wireproxy/SSH-D port). This literal URL IS stored (in the persona's own home, see decision 5).

5. **Persona egress config lives in the PERSONA's own mode-700 home, not the login user's — it is the ORDINARY `config.json`, byte-identical to v1's, just resolved in the persona's home.** There is NO new persona-config format and NO schema addition: it is the same `~/.anon-pi/config.json` anon-pi already reads (`{ proxy, llm, defaultMachine, projects?, hardened? }`), living at `~<account>/.anon-pi/config.json`, with the persona's egress in the ordinary `proxy` field. This "just works" from v1's own mechanic: anon-pi re-execs into the persona account via `sudo -u <account> -i` FIRST, so `$HOME` becomes the persona's home, `resolveAnonPiHome` resolves `~/.anon-pi` relative to THAT `$HOME`, and `readJsonConfig` reads the persona's OWN config — so the `proxy` it finds IS that persona's egress. Because the re-exec happens before anything touches config, everything downstream (config, proxy, machines, sessions) is already the persona's purely by running as that account; no special path, no second file type, no launch-time composition. The FULL literal proxy URL is stored: a Tor persona stores `proxy: "socks5h://<account>:x@127.0.0.1:9050"` (the account name AS the SOCKS-isolation username, WRITTEN into the file), a BYO persona stores `proxy: "socks5h://<host:port>"`. `persona add` COMPOSES the Tor URL once at creation (injecting the account name as the isolation token when Tor is detected + chosen); after that it is a plain `proxy` string read like any v1 config. Writing the persona name into this file is NOT a leak: the file sits inside the account's OWN mode-700 home (the login user / a host agent cannot read across the DAC wall), and the name is already self-evident there (the home is `~<account>/`, `whoami` returns it) — the name-not-in-logs principle governs argv/history/audit-log surfaces the login user CAN see, not a file locked inside the persona's home. Either way the login user's workspace holds no persona name or endpoint.

6. **An endpoint-only, no-username "used SOCKS" list in the LOGIN-user config — advisory, may go stale.** To help avoid two BYO personas accidentally sharing one endpoint (which would link them), the login-user `~/.anon-pi/config.json` holds a plain LIST OF ENDPOINTS already chosen for SOME persona — endpoints ONLY, NO persona names, NO usernames. Because it carries no correlating info (a host agent learns only "some SOCKS ports are in use", which `ss` already shows), it is safe OUTSIDE the DAC wall. Semantics: at BYO `persona add`, anon-pi shows candidate endpoints and MARKS the used ones, but the user may still pick one (their responsibility); on confirm, the chosen endpoint is appended (dedup). It is ADVISORY and may go STALE (a deleted persona's endpoint lingers; anon-pi does not read persona homes to keep it authoritative — that would breach the DAC wall). The **Tor endpoint is EXEMPT** (every Tor persona shares `:9050` by design; it must never be flagged used). This is a convenience against accidental reuse, NOT enforcement of non-reuse.

7. **`persona add` command provisions a persona; name asked interactively (same name-not-in-logs rule).** A command (e.g. `anon-pi persona add`) creates a persona: ASK the name interactively (typed, not in argv) by default, with an opt-in `--name <name>` for scripts. It runs the same two-tier flow as v1's init hardening, parameterized by the account: Tier 1 (rootless: set the persona's `ANON_PI_HOME` into its tree, chmod 700) and Tier 2 (root: create the account, subuid/subgid, linger, scoped sudoers) EMITTED as a reviewable script. Egress is chosen here (Tor-detected or BYO).

8. **Tier-2 provisioning runs INSIDE a root shell entered FIRST, via a NEUTRALLY-NAMED script — to keep persona names out of the auth/command audit log.** Instead of per-line `sudo <cmd>` (each of which logs e.g. `useradd anon-alice` — the persona name — into `/var/log/auth.log`/journald with your login name + timestamp), the user ENTERS a root shell once (`su -`, or `sudo -i`) and runs the generated script there. The single `su -`/`sudo -i` carries no persona name; the `useradd`/etc. inside the root shell are not individually sent to the sudo audit log. The script FILENAME is neutral (no persona name — it is on disk, greppable). HONESTY CAVEAT (state in docs): this is a REDUCTION, not elimination — the name unavoidably remains in `/etc/passwd`, `/etc/subuid`, `/etc/sudoers.d/…`, and `~<account>/`; root can always enumerate personas. It defends the AUDIT/HISTORY trail (sudo logs, shell history), consistent with the discoverability threat model, not against root forensics.

9. **The self-re-exec loop guard becomes "am I the TARGET persona?"** v1's guard was `os.userInfo().username === 'anon'`. With N accounts, the guard is "am I already running as the SELECTED persona account?" — so a login-user invocation re-execs into the chosen persona, and a process already running as that persona does not loop. (A persona is never auto-redirected to a DIFFERENT persona.)

## Problem Statement

v1 (prd `hardened-dedicated-account-deployment`, ADR-0006) runs ALL anonymized work under ONE shared `anon` account. That gives a single DAC wall between your login user and your anonymized work, but every persona-like activity shares one home, one netcage store, one login name, and — critically — ONE egress. Two lines of work that should be UNLINKABLE from each other (e.g. two personas that each have their own email/identity) are, under v1, trivially linkable: same account, same exit IP. The single-account model has no notion of separate personas.

The most important missing isolation is EGRESS: if `anon`'s two activities exit through the same proxy/circuit, a network observer links them by exit IP regardless of the DAC wall (which only isolates on-host data, "an issue for a tool to run in"). Real persona separation needs each persona to have its own exit.

## Solution

Generalize the single `anon` account into N dedicated PERSONA accounts, each with its own mode-700 home, session history, netcage store, AND its own fail-closed socks5h egress — so distinct personas are unlinkable both on-host (DAC) and on the network (separate exits). `anon` stays the default persona so nothing about v1 breaks.

From the user's perspective:

- **Create a persona:** `anon-pi persona add` asks (typed) for a name and an egress. If Tor is running it offers Tor multi-persona (an independent auto-Tor circuit per persona, keyed on the account name as the SOCKS-isolation username, zero setup); otherwise you give a socks5h endpoint you run. It prints a reviewable root script; you enter a root shell (`su -`/`sudo -i`) and run it, keeping the persona name out of your audit log.
- **Use a persona:** run anon-pi; on a multi-persona install it ASKS which persona (you type it, or accept `anon`), then re-execs into that account and launches with THAT persona's proxy. `--as <name>` skips the prompt when you accept the name may be logged.
- **Isolation:** each persona's work sits behind its own account's mode-700 wall (unreadable by your login user OR other personas), and each persona's traffic exits through its own proxy/circuit, so two personas never share an exit IP. Fail-closed per persona: no proxy resolved → refuse.

anon-pi implements no privilege-switching (it spawns `sudo`/`su`), never silently sudo's (Tier 2 is a human-run root script), and never weakens netcage's forced egress (still one forced socks5h per launch; this feature only picks WHICH one per persona). Persona IDENTITY (email, git config, credentials) is NOT anon-pi's concern — it is configured inside the persona's home (the container/home config).

## User Stories

1. As an operator, I want multiple dedicated persona accounts (not one shared `anon`), so that two personas are unlinkable from each other on-host, not just from my login user.
2. As an operator, I want each persona to exit through its OWN proxy/circuit, so that a network observer cannot link two personas by a shared exit IP (the isolation that matters most).
3. As an operator, I want a running Tor to give each persona an automatic, independent, self-expiring circuit keyed on the account name, so that per-persona network isolation costs no setup and near-zero resources.
4. As an operator with no Tor (or specific needs), I want to give a persona its own bring-your-own socks5h endpoint, fail-closed, so that its egress is still isolated and never guessed.
5. As an operator, I want to be reminded which BYO endpoints I have already used (endpoints only, no persona names), so that I do not accidentally put two personas on one exit — while trusting me to override.
6. As an operator, I want to TYPE a persona's name at an interactive prompt (with `--as` only as an opt-in), so that persona names never land in my shell history / process args / audit logs.
7. As an operator, I want `anon-pi persona add` to provision a persona via a reviewable root script I run inside a root shell I enter first, so that creating a persona keeps its name out of the sudo/command audit log (a reduction, honestly not elimination).
8. As an operator, I want each persona's egress config kept inside that persona's own mode-700 home, so that my login user's workspace never holds any persona's name or endpoint.
9. As an operator, I want `anon` to remain the default persona, so that my existing v1 hardened install keeps working with no change.
10. As an operator, I want the docs to state plainly that persona names are unavoidably in system files (passwd/subuid/sudoers) and that this defends the audit/history trail, not root forensics, so that I do not over-trust the name-hiding.
11. As an operator, I want persona identity (email/git/credentials) to be MY concern configured in the persona's home, not anon-pi's, so that anon-pi stays a launcher and the persona is whatever I make it inside its jail.

## Out of Scope

- **Persona identity/credentials (email, git config, keys).** Configured inside the persona's home (container/home config), not by anon-pi. anon-pi provides the isolated account + workspace + egress; the persona is what you configure in it.
- **GUI apps in a persona.** A separate concern (mostly an image concern: a virtual-screen + VNC image), captured in `work/notes/ideas/gui-apps-via-virtual-screen-image.md`. Composes with a persona but does not gate this PRD.
- **Egress rotation / per-request IP changing.** No rotation knob. A persona = one endpoint; Tor's natural ~10-min circuit churn gives coarse per-persona drift for free, which suffices. Aggressive/per-request rotation is deliberately NOT built (it can worsen unlinkability and true per-request rotation does not cleanly exist).
- **Enforced non-reuse of BYO endpoints.** The used-endpoint list is ADVISORY (decision 6); anon-pi does not (and by DAC design cannot) authoritatively track which persona uses which endpoint. A root-owned, endpoint-only, mode-600 enforcement file is a possible FUTURE enhancement, not v1.
- **Hard containment / defeating a root or blanket-sudo host agent.** Unchanged from v1: this is a discoverability + network-unlinkability boundary against an UNPRIVILEGED host agent. Root defeats the on-host half; documented, not defended.
- **anon-pi managing/launching proxies.** anon-pi composes a per-persona socks5h URL (Tor-isolation-username or BYO); it does NOT run or supervise proxy processes (you run Tor / your wireproxy).
- **A standalone workspace-migration/import into a persona.** Still deferred (idea `harden-command-with-import`); creating a persona starts it fresh.

## Further Notes

- **Supersedes ADR-0006's single-account decision.** ADR-0006 said "a single dedicated account named `anon`". This PRD generalizes to N persona accounts with `anon` as default; the tasking should write a new ADR that supersedes/extends ADR-0006 (account name is now per-persona; `ANON_ACCOUNT` becomes the DEFAULT, not the only, name; the loop guard, the Tier-2 generator, the preflight, and the config marker all parameterize by account).
- **Reuses the v1 machinery, parameterized by account.** The v1 pure pieces (`shouldRedirectToAnon`, `buildAnonSudoArgv`/`buildAnonSuFallback`, `buildTier2ProvisioningScript`, `evaluateHardenedPreflight`, `planHardeningStep`) already take the account/paths as inputs; multi-persona mostly threads a chosen persona through them + adds the persona registry-of-endpoints + the typed selection prompt + the Tor-isolation-username derivation. The self-re-exec, never-silent-root, and fail-closed-egress invariants are unchanged in shape.
- **Tor isolation mechanism (for the tasking):** `IsolateSOCKSAuth` is Tor-default; distinct SOCKS username ⇒ distinct circuit/exit; the password is an ignored placeholder; idle circuits expire at `MaxCircuitDirtiness` (~600s) and rebuild on demand; cost per persona is negligible (circuit state in the single daemon). `persona add` injects the account name as the SOCKS username when it composes the persona's `proxy` URL at creation time (stored literally in the persona's own config.json); no Tor config change is required, and launch just reads the ordinary `proxy` field.
- **Netcage is unchanged.** No netcage change is needed: it already forces one socks5h endpoint per launch, fail-closed. Multi-persona just supplies a different endpoint per persona. (Contrast v1, which also needed only netcage's uid-scoped store, still the case here.)
