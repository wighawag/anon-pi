## What to build

netcage renamed its split-tunnel flag `--allow-direct` -> `--allow` and made it PORT-MANDATORY (the all-ports / bare-IP form was removed as a deanonymization leak), shipped in **netcage v0.12.0**. anon-pi composes the launch argv with `--allow-direct <llm>`, so against netcage >= 0.12.0 it now fails with a raw netcage `unknown flag "--allow-direct"`. Adopt the new flag and require netcage >= 0.12.0, telling the user to upgrade netcage when they are below it. This is a deliberate BACKWARD-INCOMPATIBLE bump: anon-pi supports netcage >= 0.12.0 only (no dual-flag support, no version-conditional argv).

Two coupled changes:

1. **Rename the composed argv `--allow-direct` -> `--allow`.** The `llm` direct target is unchanged (still an exact `host:port`, which netcage's now-port-mandatory `--allow` requires and anon-pi already supplies via `hostPortKey(llm)`). The two argv-composition sites are the `run` RunPlan (`packages/anon-pi/src/anon-pi.ts`, the `netcageArgs.push('--proxy', proxy, '--allow-direct', directTarget)` line) and the `container enter` -> `netcage start` args (`packages/anon-pi/src/cli.ts`, the `startArgs` array). Both become `--allow`. No other behaviour changes: it is still the ONE direct hole, always present alongside `--proxy`, never stripped or duplicated.

2. **Bump the netcage version floor to 0.12.0 AND enforce it at LAUNCH time (not only in hardened onboarding).** Today `NETCAGE_MIN_VERSION` (`packages/anon-pi/src/anon-pi.ts`) is `0.11.0` and is checked ONLY in `evaluateHardenedPreflight` (the hardened onboarding step). But `--allow` is on EVERY launch (hardened or not), so a non-hardened user on netcage < 0.12.0 would otherwise hit a raw `unknown flag "--allow"` from netcage with no anon-pi guidance. So:
   - Bump `NETCAGE_MIN_VERSION` -> `0.12.0`. Update its doc comment: the floor now ALSO enforces the `--allow` flag (netcage v0.12.0's `--allow-direct` -> `--allow` rename), in addition to the existing uid-scoped-store reason (0.11.0). Keeping ONE named constant is the contract; do not scatter the literal.
   - Update `netcageVersionRemediation` so the too-old/absent message reflects that `--allow` (renamed from `--allow-direct` in netcage v0.12.0) is why the floor moved, alongside the uid-scoped store: "upgrade netcage to >= 0.12.0" stays the remediation, but the WHY line must mention the flag rename so the message is honest about the new requirement.
   - **Add a LAUNCH-TIME netcage-version gate** at the common spawn seam (`spawnNetcage` / `spawnNetcageWatch` in `packages/anon-pi/src/cli.ts`), so a jail-entering launch on netcage < 0.12.0 (or absent/unparseable) prints the remediation and exits non-zero BEFORE spawning netcage with the new flag. Reuse the existing pure helpers (`probeNetcageVersion`, `netcageVersionSatisfies`, `netcageVersionRemediation`) — do NOT duplicate the parse/compare logic. Keep it fail-loud (an absent or unparseable version is a FAILURE, never a silent pass, matching the hardened check's discipline). Scope the gate to launches that actually enter/needs the jail with the new flag (run / enter / start / the watch path); a read-only verb that does not carry `--allow` (e.g. `--version`) must not be gated.

3. **Doc / comment / help coherence sweep.** Update the `--allow-direct` references across `packages/anon-pi/src/anon-pi.ts`, `packages/anon-pi/src/cli.ts` (including the `HELP` string's `--allow-direct to a LAN model is VM-boundary-sensitive` line), the README (root `README.md`, which is copied into `packages/anon-pi/README.md` by the `copy-readme` build step — edit the ROOT), and ADR-0005 (`docs/adr/0005-container-noun-durable-boxes.md`) to `--allow`, mirroring netcage's own rename. Where a comment explains the composed argv, say `--allow`. Historical done-task records in `work/tasks/done/` are the PAST and are NOT edited (they describe what was built at the time).

## Acceptance criteria

- [ ] The composed netcage argv uses `--allow <host:port>` (not `--allow-direct`) on every launch path: `run` (RunPlan) and `container enter` (`netcage start`). The direct target is unchanged (exact `host:port`), still the ONE hole, always present with `--proxy`.
- [ ] `NETCAGE_MIN_VERSION` is `0.12.0`, kept as the single named constant; its doc comment and `netcageVersionRemediation` name BOTH reasons the floor requires it (uid-scoped store from 0.11.0, and the `--allow` flag rename from 0.12.0).
- [ ] A jail-entering launch on netcage `< 0.12.0` (or absent, or unparseable version) is refused at LAUNCH time with the netcage-version remediation and a non-zero exit, BEFORE anon-pi spawns netcage with `--allow`. This holds for a NON-hardened install too (the gate is at the spawn seam, not only in hardened onboarding). Fail-loud: absent/unparseable is a failure, never a silent pass. Reuses the existing pure version helpers (no duplicated parse/compare).
- [ ] A launch on netcage `>= 0.12.0` proceeds normally (the gate is a floor, not a pin: 0.12.0, 0.12.1, 0.13.0 all pass).
- [ ] The hardened onboarding preflight still enforces the (now 0.12.0) floor, with its remediation text consistent with the launch-time one.
- [ ] No lingering `--allow-direct` in anon-pi's OWN operator-facing surface (composed argv, README, help text, ADR-0005, live source comments). Historical `work/tasks/done/` records are left untouched.
- [ ] A `.changeset/*.md` with `'anon-pi': minor` documents the breaking netcage-floor bump + flag adoption.
- [ ] The whole `verify` is green: `pnpm format:check && pnpm changeset status --since=main && pnpm -r build && pnpm -r test`.
