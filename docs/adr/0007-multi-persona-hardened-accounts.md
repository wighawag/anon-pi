# Multi-persona hardened deployment: N dedicated `anon-<name>` accounts with per-persona fail-closed egress

## Status

Accepted. Formalizes spec `multi-persona-hardened-accounts`. **Supersedes ADR-0006** (single dedicated `anon` account): the hardened deployment is now N dedicated persona accounts `anon-<name>`, with `anon` the DEFAULT persona (uniform "persona #0", not a special-cased always-present account). ADR-0006's decisions that are UNCHANGED (self-re-exec not setuid, always-redirect, sudo password kept by default, netcage's uid-scoped store) still hold, now parameterized per persona; only its "one canonical shared account named `anon`, pinned so it can never be re-forked" decision is replaced. See ADR-0006 (carries a superseded-by note pointing here).

## Context

ADR-0006 ran ALL anonymized work under ONE shared `anon` account. That gives a single DAC wall between your login user and your anonymized work, but every persona-like activity shares one home, one netcage store, one login name, and, critically, ONE egress. Two lines of work that should be UNLINKABLE (two personas each with their own email/identity) are, under v1, trivially linkable: same account, same exit IP. The single-account model has no notion of separate personas, and the most important missing isolation is EGRESS: if two activities exit through the same proxy/circuit, a network observer links them by exit IP regardless of the on-host DAC wall (which only isolates on-host data).

The threat model is UNCHANGED from ADR-0006: this is a DISCOVERABILITY + network-unlinkability boundary against an UNPRIVILEGED host agent running as your login user, NOT hard containment. A host agent with root or blanket passwordless sudo defeats the on-host half entirely (documented, not defended). What multi-persona ADDS is the network-unlinkability half (per-persona exits) and on-host unlinkability BETWEEN personas (each behind its own account's DAC wall), not just between a persona and the login user.

## Decision

Generalize the single `anon` account into N dedicated PERSONA accounts, each its own unlinkable persona with its own mode-700 home, session history, netcage store, AND its own fail-closed egress. `anon` stays the DEFAULT persona name, so existing v1 hardened installs are simply a multi-persona install with exactly one persona named `anon` and keep working unchanged.

- **Accounts are `anon-<name>`; the user types the bare `<name>`; the default is the bare `anon`.** A user-facing bare name (`alice`) maps to the Unix account `anon-<name>` (`anon-alice`); the default persona is the prefix with an EMPTY suffix, i.e. the bare `anon` (`ANON_ACCOUNT`, `personaAccount(undefined) === 'anon'`). The bare name is validated as a conservative lowercase Unix-username suffix (`^[a-z0-9][a-z0-9-]*$`, no `anon-` double-prefix), so it is a clean label for passwd / sudoers / home-path / the Tor SOCKS-isolation username with no quoting surprises. The `anon-` prefix reveals only "this box runs anonymization tooling" to a root / `/etc/passwd` reader (already true of v1's `anon`), never persona linkage; it is the on-host surface we do not defend anyway.

- **`anon` is uniform "persona #0", not special-cased.** `anon` is created by the same `persona add` path as any other persona; there is no distinguished always-present account. The code stays one shape: a list of personas, one of which carries the default name `anon`. `ANON_ACCOUNT` is now the DEFAULT account (the empty-suffix case), NOT the only account.

- **Selection = `--as <name>`, defaulting to `anon`.** On a hardened install a launch runs as the persona named by `--as <name>` (mapped to `anon-<name>`), defaulting to `anon` when `--as` is absent. The name is a PLAIN argument: it may appear in shell history / `ps`, and that is ACCEPTED (the name is already in passwd/sudoers/home-path anyway). `--as <name>` is STRIPPED from the argv netcage / the launch grammar sees (`stripAsFlag`) yet SURVIVES into the self-re-exec (the redirect forwards the raw argv; the re-exec'd child strips it before composing the jail). A `--as` naming a persona that does not exist is an ERROR ("no persona `<name>`; create it with `anon-pi persona add <name>`"), never a silent create and never a silent fall-through to `anon`. `--version` stays local (no redirect). (A history-hygiene variant, an interactive typed prompt so the name never enters argv, was considered and DEFERRED to the idea `persona-name-history-hygiene`; see Considered Options.)

- **The self-re-exec loop guard is "am I the TARGET persona?"** v1's guard was `os.userInfo().username === 'anon'`. With N accounts the guard (`shouldRedirectToPersona`) is "am I already running as the SELECTED persona account?": a login-user invocation re-execs into the chosen persona (via `buildAnonSudoArgv` with the selected account, `sudo -u anon-<name> -i <anon-pi> …`; `su - anon-<name> -c …` the documented fallback), and a process already running as that persona does not loop. A persona is NEVER auto-redirected to a DIFFERENT persona. The no-`--as` default is byte-behaviour-identical to v1.

- **Per-persona FAIL-CLOSED egress: one socks5h endpoint per persona.** Each persona has its OWN egress; no persona shares another's exit. Fail-closed is per persona: a persona with no resolvable proxy REFUSES to launch, byte-identically to v1's `PROXY_REQUIRED_MESSAGE`, and never falls back to another persona's proxy or to none. Netcage's forced-egress invariant is UNCHANGED: still exactly one socks5h endpoint forced per launch, still never guessed. This feature only changes WHICH endpoint (per persona) and WHICH account anon-pi runs as; it never weakens the jail. There is NO rotation knob (a persona = one endpoint; Tor's natural circuit churn gives coarse per-persona drift for free).

- **Two ways to get a per-persona endpoint: Tor multi-persona (preferred) or bring-your-own SOCKS.**
  - **Tor multi-persona:** anon-pi uses the persona's ACCOUNT USERNAME as the Tor SOCKS-isolation username, composing the literal `socks5h://<account>:x@127.0.0.1:9050` (`composeTorPersonaProxy`; the password `x` is an ignored placeholder). Tor's `IsolateSOCKSAuth` (on by default) then builds a SEPARATE circuit + exit per distinct SOCKS username, so each persona gets an independent, auto-managed, auto-expiring circuit (idle circuits tear down at `MaxCircuitDirtiness` ~600s and rebuild on next use) at near-zero marginal cost. `persona add` OFFERS Tor when it detects a running Tor (a SOCKS probe, `offerTor` over the injected detection); on a TTY with neither egress flag it probes and offers automatically, or `--tor [<host:port>]` forces it.
  - **Bring-your-own SOCKS:** otherwise the user gives a socks5h endpoint they run (`--proxy <socks5h-url>`, or the interactive prompt). anon-pi prints a one-line WARNING (`PERSONA_BYO_UNIQUENESS_WARNING`): this endpoint must be UNIQUE to this persona; two personas on one BYO endpoint share an exit and become linkable, so prefer Tor. anon-pi stores NO used-endpoint registry (see Considered Options).

- **Persona egress config lives in the PERSONA's own mode-700 home; it is the ordinary `config.json`, not a new format.** The composed/entered proxy is stored literally in the persona's own `~<account>/.anon-pi/config.json` `proxy` field, byte-identical in shape to v1's config, just resolved in the persona's home. This "just works" from v1's own mechanic: the self-re-exec enters the persona account (`sudo -u <account> -i …`) FIRST, so `$HOME` becomes the persona's home, `resolveAnonPiHome` resolves `~/.anon-pi` relative to THAT `$HOME`, and `readJsonConfig` reads the persona's OWN config, whose `proxy` IS that persona's egress. No new schema, no schema marker, no launch-time composition, no launch-time re-derivation. `persona add` composes the Tor URL ONCE at creation (injecting the account name as the isolation username); after that it is a plain `proxy` string read like any v1 config. Writing the persona name into this file is NOT a leak: the file sits inside the account's OWN mode-700 home (the login user / a host agent cannot read across the DAC wall), and the name is already self-evident there. The name-not-in-logs principle governs the argv/history/audit-log surfaces the login user CAN see, not a file locked inside the persona's home. Either way the login user's own workspace holds no persona name or endpoint.

- **`anon-pi persona add <name>` provisions a persona.** It runs the same two-tier flow as v1's init hardening, parameterized by the account: Tier 1 (rootless: create the persona's mode-700 `~anon-<name>/.anon-pi` and write its own `config.json` carrying the chosen `proxy`) and Tier 2 (root: create the account, enable linger, install the scoped sudoers rule). Egress is chosen here. The flow is RESUMABLE around the root step (the Tier-1 in-home write needs the account to exist first): while the account is missing it emits Tier 2 and waits; once it exists (re-probed via `getent`) it does Tier 1. Re-running `persona add <name>` for an already-provisioned persona is an idempotent no-op (a re-check), never a duplicate or a failure. Persona IDENTITY (email/git/credentials) is explicitly OUT of scope: configured inside the persona's home.

- **Tier-2 provisioning = COPY-PASTE COMMANDS run inside a root shell entered FIRST; no script FILE is written; subid auto-allocated.** `buildTier2ProvisioningScript` now returns a block the human pastes into a root shell they enter first (`sudo -i` / `su -`): `useradd -m <account>`, `loginctl enable-linger <account>`, and the scoped sudoers rule (`<login-user> ALL=(<account>) <anon-pi>`, password KEPT by default, `visudo -cf`-validated before `install -m 0440` into the per-account file `/etc/sudoers.d/anon-pi-<account>`). anon-pi PRINTS this and NEVER executes it. There is NO generated `#!/bin/sh` script FILE (v1's `mktemp` artifact is RETIRED for BOTH the default `anon` and every persona): nothing on disk to leak the persona name, nothing to save. Running the commands inside a root shell keeps the persona name out of the sudo/command AUDIT log (the single `sudo -i` carries no name; commands typed in a root shell are not individually audited). The subuid/subgid block is AUTO-ALLOCATED by `useradd -m` (modern shadow-utils via `SUB_UID_COUNT`/`SUB_GID_COUNT`): anon-pi emits NO explicit `/etc/subuid`+`/etc/subgid` range line, which removes the range-collision math entirely and gives each of N personas a distinct block for free. The preflight still verifies a range EXISTS for the account; it just no longer asserts a specific start.

## Honesty caveat (state in docs)

The name-hiding is a REDUCTION, not an elimination. The persona name unavoidably remains in `/etc/passwd`, `/etc/subuid`/`/etc/subgid`, `/etc/sudoers.d/anon-pi-<account>`, and `~anon-<name>/`; root can always enumerate personas. Hiding the name defends the AUDIT/HISTORY trail (the sudo/command log, shell history), consistent with the discoverability threat model, NOT root forensics. Docs MUST NOT claim hard containment.

## Considered Options

- **Bare `<name>` accounts (no `anon-` prefix).** Rejected: a persona `alice` as a bare `alice` account could collide with a real system/human account; `anon-alice` is collision-safe and self-labelling in `/etc/passwd`, and the Tor SOCKS-isolation username is then the distinct namespaced `anon-<name>` automatically.
- **A generated Tier-2 script FILE (v1's shape).** Rejected/RETIRED: an on-disk `#!/bin/sh` file leaks the persona name and is one more thing to save/run; copy-paste commands into a root shell entered first need no file and keep the name out of the audit log.
- **An explicit `/etc/subuid`+`/etc/subgid` range line (v1's `SUBID_RANGE_START`/`SUBID_RANGE_COUNT`).** Rejected/RETIRED: it forced range-collision math on anon-pi. `useradd -m` auto-allocates a free non-overlapping block per account, so N personas never collide and anon-pi carries no range math.
- **Storing the proxy behind a schema marker / re-deriving it at launch.** Rejected: the proxy is stored LITERALLY in the persona's own `config.json` `proxy` field. It is not a leak (behind the persona's DAC wall) and it lets launch read an ordinary v1 `proxy` with no per-launch composition and no second config format.
- **A used-SOCKS-endpoint registry to enforce BYO uniqueness.** Rejected as over-built: for Tor there is no collision (every Tor persona shares `:9050` and Tor isolates by username, which IS the design); for BYO, anon-pi cannot read across personas' DAC walls to track this authoritatively, so it WARNS at `persona add` and leaves uniqueness to the operator. A root-owned enforcement file is a possible FUTURE enhancement, not v1.
- **An interactive typed persona-selection prompt (name out of shell history).** Considered and DEFERRED to the idea `persona-name-history-hygiene`: the name is already in passwd/sudoers/home-path/`ps`, so protecting it only in login-shell history is a narrow, low-value benefit not worth making the primary path in v1. v1 takes the name as a plain `--as <name>` / `persona add <name>` argument.

## Consequences

- The v1 pure machinery is REUSED, parameterized by account: `shouldRedirectToPersona` (generalizing `shouldRedirectToAnon`), `buildAnonSudoArgv`/`buildAnonSuFallback` (now taking an optional `account`, defaulting to `anon` so v1 callers stay byte-identical), `buildTier2ProvisioningScript`, `evaluateHardenedPreflight`/`probeHardenedPreflight` (already account-parameterized and range-existence-based), and `planHardeningStep`. New pure surface: `personaAccount`/`personaName`/`validatePersonaName`, `resolvePersonaSelection`/`stripAsFlag`, `composeTorPersonaProxy`/`offerTor`, and `parsePersonaArgs`/`planPersonaAdd`.
- **Netcage is UNCHANGED.** It already forces one socks5h endpoint per launch, fail-closed, and its uid-scoped store (netcage ADR-0017, floor `>= 0.11.0`) auto-scopes per account; multi-persona just supplies a different endpoint per persona and runs as a different account. anon-pi still sets NO `NETCAGE_GRAPHROOT`.
- The default path is behaviour-preserving: no `--as`, an existing `anon` install redirects to `anon` and launches exactly as before (regression-guarded).
- Persona IDENTITY (email, git config, credentials) is NOT anon-pi's concern; it is configured inside the persona's home. anon-pi remains a launcher that provides the isolated account + workspace + egress.
